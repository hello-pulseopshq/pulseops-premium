# PulseOps Concept Translation Standard

Version: 1.0

Status: Foundation

---

# Purpose

Concept Translation is the bridge between creative exploration and visual design.

Its purpose is to preserve the creative intent of the selected concept before mockup creation begins.

Concept Translation exists because editorial concepts and visual designs are not the same thing.

A concept explains the idea.

A mockup expresses the idea.

Concept Translation ensures the original idea survives that journey.

# Philosophy

Creative intent should be frozen before visual design begins.

Desktop and mobile mockups should solve visual problems.

They should not reinterpret narrative or invent a new concept.

Concept Translation exists to eliminate that ambiguity.

---

# Principles

## Principle 1

Concept Translation preserves intent.

It does not create layouts.

It freezes the creative direction before visual design begins.

---

## Principle 2

One concept.

One translation.

Do not merge multiple concepts during mockup creation.

If multiple concepts are desirable, they should be intentionally combined before Concept Translation is written.

---

## Principle 3

Creative decisions belong here.

Visual decisions belong in mockups.

Implementation decisions belong in Blueprint and Annotated Mockups.

---

## Principle 4

Translation documents purpose.

Not pixels.

Avoid discussing:

- spacing
- font sizes
- margins
- CSS
- implementation
- components

---

## Principle 5

The memory anchor is sacred.

Every later production stage should strengthen the same memory anchor.

Nothing should dilute or replace it.

---

## Principle 6

Desktop and mobile inherit the same creative intent.

They may have different compositions.

They must preserve the same chapter identity.

# Inputs

Concept Translation begins only after one concept has been approved.

Required inputs:

- Approved concept
- Chapter Identity
- Review findings
- PulseOps Design System
- Architecture
- Narrative
- Design Languages

If the concept has not been approved:

Stop.

Do not begin Concept Translation.

---

# Outputs

Concept Translation produces a frozen creative direction.

It becomes the source of truth for all visual design work.

---

# Standard Structure

Every Concept Translation should contain the following sections.

---

# 1. Chapter

Example

Community Confidence

---

# 2. Concept Name

Record the exact selected concept.

Example

Morning Index

or

Morning Index × Understated Consensus

---

# 3. Core Idea

One paragraph.

Summarise the concept.

Explain the central editorial idea.

This should be understandable by someone who has never seen the concept exploration.

---

# 4. Buyer Question

Document the single buyer question.

Do not modify it.

Example

> Will people like me actually stick with this?

---

# 5. Narrative Responsibility

Describe the chapter's role in the editorial journey.

Explain:

Previous chapter

↓

Current chapter

↓

Next chapter

The narrative responsibility should never overlap neighbouring chapters.

---

# 6. Memory Anchor

Define exactly what the visitor should remember after leaving the chapter.

Only one memory anchor should exist.

Good examples:

- Morning Index
- Editorial Ingredient Gallery
- Human Portrait
- Ritual Timeline

Poor examples:

- Premium
- Ingredients
- Trust

---

# 7. Emotional Goal

Describe how the visitor should feel after reading the chapter.

Examples:

Calm confidence

Curiosity

Reassurance

Belonging

Momentum

Reflection

---

# 8. Desktop Composition Philosophy

Describe the design philosophy.

Do not describe layout measurements.

Instead explain:

- visual ownership
- hierarchy
- reading rhythm
- editorial pacing
- dominant visual moment

Example

Typography owns the experience.

The Morning Index is the dominant visual moment.

Supporting copy reinforces rather than competes.

---

# 9. Mobile Composition Philosophy

Describe how the same concept behaves on mobile.

Do not simply state:

"Stack desktop."

Instead explain:

- rhythm
- pacing
- hierarchy
- editorial experience

Mobile should preserve the same memory anchor while behaving naturally on a smaller screen.

---

# 10. Photography Ownership

Document photography's responsibility.

Examples:

Photography owns the chapter.

Photography supports typography.

Photography is intentionally absent.

Photography provides atmosphere but not narrative ownership.

---

# 11. Typography Ownership

Describe typography's responsibility.

Examples:

Typography is the dominant visual owner.

Typography behaves as editorial navigation.

Typography reinforces documentary credibility.

Typography expresses emotional authority.

---

# 12. Surface Ownership

Describe the intended material feeling.

Examples:

Quiet editorial canvas.

Museum-like restraint.

Paper-like documentary.

Minimal intervention.

Do not define implementation.

---

# 13. Interaction Philosophy

Describe how interaction supports the chapter.

Examples:

Minimal.

Reading-first.

Hover reveals secondary information.

Touch interactions preserve calm.

Interaction should never become the memory anchor.

---

# 14. Motion Philosophy

Describe motion behaviour.

Examples:

Motion should feel nearly invisible.

Movement reinforces reading rhythm.

Motion never competes with typography.

Reduced-motion users receive the same narrative.

---

# 15. Merchant Experience

Document what merchants should be able to customise.

Examples:

- copy
- photography
- CTA
- ordering
- colours
- spacing

Document what should remain fixed.

Examples:

- chapter structure
- editorial hierarchy
- memory anchor
- composition philosophy

---

# 16. Cross-Vertical Adaptation

Explain how the concept scales.

Demonstrate adaptation for at least:

- Supplements
- Skincare
- Coffee
- Electronics
- One additional PulseOps vertical

Adapt through:

- photography
- copy
- products
- context

Do not change:

- narrative role
- composition philosophy
- buyer question
- memory anchor

---

# 17. Things That Must Never Change

This section protects the concept.

Examples:

The Morning Index must remain the dominant memory anchor.

Community must be communicated through shared routine rather than statistics.

Typography must remain the primary visual owner.

No dashboard aesthetics.

No testimonial cards.

---

# 18. Things Intentionally Left Open

Identify areas that designers may explore.

Examples:

Exact photography.

Typography styling.

Spacing.

Editorial cropping.

Micro-interactions.

These decisions belong to mockup creation.

---

# Approval Criteria

Concept Translation is approved only when:

- One concept is clearly represented.
- One buyer question is preserved.
- One memory anchor is defined.
- Desktop and mobile philosophy are documented.
- Creative ownership is clear.
- No implementation guidance exists.
- No layout measurements exist.
- No CSS or engineering decisions exist.

---

# Anti-Patterns

Avoid:

- Creating mockups inside the Concept Translation.
- Discussing CSS.
- Discussing Liquid.
- Discussing implementation.
- Defining spacing or dimensions.
- Introducing multiple memory anchors.
- Changing the approved concept.
- Solving responsive layout problems.
- Designing desktop.
- Designing mobile.

Those belong to later production stages.

---

# Definition of Done

A Concept Translation is complete when a designer can create desktop and mobile mockups without needing to reinterpret the original concept.

If two designers independently create mockups from the same Concept Translation, both designs should clearly express the same memory anchor, buyer question, and editorial intent—even if the visual execution differs.

Concept Translation succeeds when creativity becomes intentional rather than ambiguous.
